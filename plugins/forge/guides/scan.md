# 竞品扫描方法论

> 目的：回答一个问题——**别人已经怎么解决这个痛点了？** 只摆事实，带证据，不做产品判断。

## 流程

1. 读需求（标题 + 痛点 + 谁有）和挑战记录里的事实缺口（如有）
2. 用 `web_search` 搜 2-4 轮，覆盖：
   - 直接竞品：同目标的成品/工具（中文 + 英文关键词各搜一轮）
   - 替代方案：用户现在的土办法（Excel/微信群/手动流程也算方案）
   - 反向验证：搜「{领域} 失败 / 没人用 / 弃坑」——死掉的先行者是最好证据
3. 整理竞品盘点表（**每条必须带来源链接**，搜不到就写「未找到」，不许编）：

```
| 方案 | 怎么解决的 | 证据 | 与我们的差异点 |
|------|-----------|------|---------------|
| （名称）| （一句话）| （链接）| （一句话）|
```

4. 收尾三句话：
   - 市面覆盖度：红海 / 有空白 / 几乎无人做
   - 无人做的原因（如有）：是真空白，还是伪需求/做不起来？
   - 对需求的影响：支持 / 削弱 / 需要重新定义差异点

5. **落库**：把盘点表 + 三句话追加进快筛结论，调 `forge.triage` 更新 triage 字段（status 不变，传当前原值）

## 纪律

- 最多 4 轮搜索。扫不到足够信息就写「证据不足」，不硬凑
- 来源链接必须真实（来自搜索结果），**禁止编造 URL**
- 这一环不做「要不要做」的判断——那是挑战和裁决的事
